## Protocol Implementation

### Client → Server
- `Hello` - Player identity (UUID, name, track_id)
- `WheelDrawing` - Initial wheel submission
- `WheelSwap` - Mid-race wheel swap
- `Ping` - Latency measurement

### Server → Client
- `Welcome` - Room assignment with player list
- `RaceStart` - Countdown with start time
- `State` - 30Hz authoritative state broadcasts
- `PlayerJoined` / `PlayerLeft` - Lobby events
- `RaceFinished` - Final times and ranks
- `Pong` - Ping response

## Architecture (per plan §Multiplayer 13)

1. **Sticky sessions via rooms, not LB**
   - Each race = room keyed by `race_id` (UUID)
   - Redis `HSET race:{id} pod {pod_ip}` for routing
   - Ingress cookie-based sticky session as fallback

2. **Authoritative sim**
   - Pod runs same WASM physics module as client
   - 30Hz fixed step (TODO: actual WASM integration)
   - Broadcasts `{racer_id, x, y, angle, t}` at 30Hz

3. **State sync rate: 30Hz**
   - ~200 bytes per racer per tick
   - Client-side interpolation (same as ghost system)

4. **Reconnect semantics**
   - Rooms persist in Redis for 10 minutes
   - Clients can reconnect with player_uuid

5. **Matchmaker upgrade**
   - Lobby pools partitioned by track and bucket
   - 8-second timeout before starting with ghosts
   - Ghosts fill empty slots from same bucket
